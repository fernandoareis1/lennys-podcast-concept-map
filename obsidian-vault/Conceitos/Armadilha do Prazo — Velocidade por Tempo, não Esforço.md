# Armadilha do Prazo — Velocidade por Tempo, não Esforço
**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Daniel Lereya]]

## Ideia central
Planejar de baixo para cima (bottom-up) gera protecionismo: o time lista riscos, adiciona buffers, e projetos que deveriam levar semanas acabam levando um ano. A solução é **definir o deadline primeiro** e trabalhar o escopo para dentro desse prazo.

Daniel Lereya chama isso de "colocar armadilhas para si mesmo definidas por tempo, não por esforço."

**O padrão do planejamento bottom-up:**
1. Time começa a planejar uma feature
2. Riscos são listados → cada risco gera mais escopo
3. Preocupações são levantadas → mais conteúdo adicionado
4. Resultado: "ok, vai para daqui a 2 anos"

**O padrão da armadilha de prazo:**
1. Defina o prazo com antecedência (ex: "queremos anunciar isso no próximo earnings")
2. Pergunte: "o que conseguimos shipar nesse prazo?"
3. Conversa muda imediatamente: foco no core do valor, não nas bordas
4. Time fica mais agudo, mais focado, e entrega mais rápido

**Por que mais tempo = pior produto:**
- Mais tempo cria mais perguntas que você responde sozinho, sem o usuário
- Mais tempo cria mais suposições inventadas sobre o que o usuário quer
- O medo de lançar ("e se não gostarem?") cresce à medida que o projeto fica maior
- A primeira versão perfeita é uma ilusão que mais tempo sustenta

**O feedback correto da primeira versão:**
"Isso está prematuro" ou "estou sentindo falta de X e Y" = **ótimo resultado**. Significa que você construiu o núcleo sem gordura e descobriu o próximo passo com clareza. "Que produto incrível!" na primeira versão = você construiu coisas que não eram o núcleo.

**Aplicação em diferentes escalas:**
- Startup: "temos earnings/demo em 3 semanas, o que cabe?"
- Empresa grande: "quero anunciar no kickoff anual, o que precisamos ter?"
- Time pequeno: "duas semanas, o que é o MVP real?"

## Como aplicar
1. **Decida o deadline antes do escopo:** para o próximo projeto, anuncie o prazo para o time antes de discutir o que vai dentro; observe como a conversa muda
2. **Corte escopo, não qualidade:** dentro do prazo fixo, o que sai não é qualidade — é itens do backlog que podem esperar para a v2
3. **Embrace o feedback "prematuro":** quando usuários dizem "está prematuro, falta X", comemore com o time; vocês descobriram o próximo passo real de uma forma que meses de planejamento não teriam revelado
4. **Use eventos externos como âncoras:** earnings, conferências, lançamentos de parceiros, kickoffs internos — qualquer data fixa no calendário é uma oportunidade para criar a armadilha
5. **Identifique quando está em modo protecionista:** se o time está passando mais tempo listando riscos do que construindo, é sinal de que você está no modo bottom-up; reset com uma data fixa
