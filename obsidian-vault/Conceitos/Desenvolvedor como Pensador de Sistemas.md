# Desenvolvedor como Pensador de Sistemas
**Fluxo:** [[09 - Liderança & Times]] · **Tema:** [[Liderança — Modelos de time]] · **Camada:** L3
**Tipo:** Mindset · **Fontes:** [[Inbal Shani]]

## Ideia central
À medida que AI assume a escrita de código mecânico, o **locus de valor do desenvolvedor se desloca para cima**: de escrever funções para entender sistemas, arquitetura e experiências conectadas. Esse nível de pensamento existia antes — mas principalmente em desenvolvedores seniores. AI democratiza o acesso a ele.

**A evolução histórica das camadas de abstração:**
- **C** → programador controla cada alocação de memória
- **Java** → bibliotecas e frameworks absorvem complexidade
- **Python** → outra camada; ainda menos mecânica
- **AI** → próxima camada de abstração; libera atenção para pensar no sistema como um todo

Isso **não elimina** o desenvolvedor — muda o que o desenvolvedor faz. Quem sabia programar em C quando Java chegou não ficou desempregado; atualizou onde aplicava seu esforço.

**O impacto nos desenvolvedores juniores:**
Hoje, espera-se que júniors comecem escrevendo código simples — porque ainda estão aprendendo sintaxe. Com AI cuidando da mecânica, eles podem **desde o primeiro dia** entender o sistema, o ambiente e o produto que estão construindo. O processo de formação acelera porque o gargalo (aprender a escrever código) sai do caminho.

**O que permanece humano:**
- Entendimento do sistema e das interdependências
- Julgamento sobre arquitetura e trade-offs de design
- Colaboração e comunicação entre personas (produto, engenharia, negócio)
- Criatividade e inovação — a "centelha criativa"
- Compreensão do que o produto significa para o usuário final

## Como aplicar
1. **Redefina onboarding de eng júnior**: em vez de começar com "escreva este módulo", comece com "entenda este sistema — mapa de dependências, decisões de arquitetura, casos de uso". AI ajuda com as partes mecânicas.
2. **Mude o critério de code review**: menos "está correto?" (AI verifica) e mais "está bem arquitetado? Faz sentido no contexto do sistema?"
3. **Crie exposição de sistema desde cedo**: pair sessions entre seniores e júniors focadas em sistema, não em código. Peça ao júnior que explique "como isso conecta com X?"
4. **Invista em testing automatizado**: como AI gera mais código, testing se torna o gargalo. Área underhyped: AI-driven testing (unit, load, security, penetration) para validar o que AI produz.
5. **Selecione desenvolvedores por pensamento sistêmico**: em seleção, pergunte sobre decisões de arquitetura, não apenas sobre sintaxe. A habilidade de "pensar o grande sistema" será o diferencial.

## Insights por autor
### [[Inbal Shani]]
- "It's no longer just the actual code writing, it's really evolving your thinking to the big picture, to the connected experience, to connected systems, which today we just find it more in the world of more senior developers."
- "If now there is an AI assistant that is helping them writing code, [junior devs] can spend more time from the get-go understanding the system, understanding the environment that they're building, or understanding the product — which today they don't have time because they are still learning how to code."
- Analogia das camadas: "When I was learning how to code, I used to code in C, then Java became a thing. I'm like, oh, but this is taking away my job. And then Python came. It's a matter of how you use the tools and what you use them to do what."
- Testing como underhyped: "If we're going to generate much more code, much more productive, more lines of code, more applications, then testing is becoming such a critical part. AI-driven testing — I would like to hear more about what companies are thinking about how to leverage AI to generate a larger suite of testing."
