# Exploração de Extremos para Solução Certa
**Fluxo:** [[05 - Experimentação & Dados]] · **Tema:** [[Experimentação — Aprender com pouco e qualitativo]] · **Camada:** L3
**Tipo:** Técnica · **Fontes:** [[Nan Yu]]

## Ideia central
Para fugir dos defaults cognitivos e encontrar soluções não-óbvias, o time do Linear deliberadamente constrói versões extremas de uma feature — a mais rápida possível e a mais segura possível — antes de encontrar o equilíbrio correto. Isso expande o espaço de busca e gera aprendizado real a partir de experiência vivida, não de suposições. As melhores soluções são sempre óbvias em retrospecto; o processo de chegar lá exige ultrapassar os limites para saber onde eles estão.

## Como aplicar
- Para qualquer decisão de produto, identifique o atributo central de tensão (velocidade vs. segurança; simplicidade vs. poder)
- Construa a versão mais extrema em cada direção — mesmo sabendo que provavelmente não é a resposta certa
- Lance os extremos para usuários reais o mais cedo possível para sentir (não imaginar) os problemas
- Use o aprendizado dos extremos para encontrar o ponto de equilíbrio com convicção baseada em evidência
- "A quantidade certa é demais menos um" — você precisa ultrapassar o limite para saber onde ele está

## Insights por autor
### [[Nan Yu]]
- Exemplo ao vivo com o feature de drafts: construiu versão ultra-rápida (fecha sem perguntar) → equipe sentiu insegurança real → construiu versão ultra-segura (auto-salva tudo) → usuários acumularam dezenas de drafts indesejados → chegou à solução híbrida com convicção.
- "The best solutions are always obvious in hindsight, and it's just that you have to develop a process internally to eventually find your way there."
- O problema não é falta de criatividade, mas não expandir o espaço de busca — a quarta opção correta fica invisível se você só compara três opções medianas.
