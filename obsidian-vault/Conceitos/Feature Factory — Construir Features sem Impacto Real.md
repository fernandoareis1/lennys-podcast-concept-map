# Feature Factory — Construir Features sem Impacto Real

**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Gokul Rajaram]]

## Ideia central
Uma **feature factory** é uma equipe de produto que é muito boa em entregas, mas não sabe se essas entregas mudaram o comportamento do usuário. A metáfora: "Se uma árvore cai na floresta e ninguém ouve, ela fez barulho?" — se um feature é lançado e não muda o comportamento do cliente, ele foi realmente lançado? A raiz do problema quase sempre é um founder ou PM que apresenta soluções (táticas) em vez de problemas — o que mata o empoderamento do time de engenharia e design.

## Como a feature factory se instala
1. O founder assume que sabe o que o cliente quer
2. Passa táticas ao time ("construa um app iOS") em vez de problemas ("aumente de 2 para 5 sessões por semana")
3. Engenheiros se habituam a executar, não raciocinar
4. Quando o primeiro PM chega, ele herda o mesmo padrão e perpetua o ciclo

## O antídoto: apresentar problemas, não soluções
O líder de produto deve:
1. Articular claramente o comportamento do cliente que precisa mudar e por quê
2. Trazer dados de suporte (NPS, retenção, entrevistas)
3. Convidar o time a explorar 3+ abordagens para o mesmo problema
4. Discutir o ROI de cada abordagem antes de escolher
5. Medir o impacto pós-lançamento com critérios definidos antes do lançamento

## Sinal de feature factory vs. time empoderado
| Feature factory | Time empoderado |
|---|---|
| "Construímos X features este trimestre" | "Mudamos Y comportamento do cliente em Z%" |
| Engenheiros esperam especificação | Engenheiros propõem abordagens para o problema |
| PM define o "como" | PM define o "porquê" e o "quanto" |
| Roadmap de outputs | Roadmap de outcomes |
