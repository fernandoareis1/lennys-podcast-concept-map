# Forward Deployed Engineering — Produto Nasce do Campo
**Fluxo:** [[02 - Estratégia & Visão]] · **Tema:** [[Estratégia — Estratégia de produto]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Nabeel S. Qureshi]]

## Ideia central
Em vez de vender um produto pronto e depois adaptar, a Palantir envia engenheiros diretamente para dentro dos clientes — com mesa própria no escritório do cliente, segunda a quinta — para construir soluções reais no campo. Engenheiros forward deployed (FDE) têm poderes para criar produto novo, não apenas configurar o existente. O que funciona se abstrai e vira plataforma. A sequência: entenda o problema → construa a solução → abstraia para produto.

## Como aplicar
1. **Mande pessoas reais (engenheiros) para o campo**: não soluções pré-embaladas, não apenas consultores. FDEs precisam poder construir produto novo — não só configurar.
2. **Dê permissão para criar**: a distinção-chave da Palantir era: FDE pode construir produto completamente novo se o cliente precisar. "Soluções architect" que não constroem não têm o mesmo efeito.
3. **Aprenda a linguagem do cliente**: após semanas dentro de uma fábrica ou governo, o engenheiro fala a língua do domínio. O produto resultante reflete isso.
4. **Abstraia o que funciona**: quando uma solução resolve o problema de um cliente, pergunte: qual é a versão produto disto? O Ontology da Palantir nasceu de um problema real de uma fábrica da Airbus.
5. **Controle o escopo do tempo no campo**: Palantir usava segunda a quinta no cliente; sexta era para síntese e construção de produto. Não deixe o campo virar consultoria permanente sem abstração.
