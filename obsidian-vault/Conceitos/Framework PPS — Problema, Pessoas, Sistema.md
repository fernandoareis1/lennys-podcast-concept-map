# Framework PPS — Problema, Pessoas, Sistema
**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Austin Hay]]

## Ideia central
Quando surge um desafio de sistemas, dados ou tecnologia, a reação instintiva é pular direto para a solução — "preciso de acesso admin", "vamos trocar de ferramenta". O **Framework PPS** força um recuo: antes de qualquer decisão de sistema, passe por três perguntas em sequência: (1) **Problema** — o que exatamente essa pessoa está tentando resolver? Qual é o problema discreto? (2) **Pessoas** — quem está envolvido? Quem precisa de permissão? Quem será afetado? Há fatores confusos que surgem dos envolvidos? (3) **Sistema** — só então: qual sistema ou ferramenta resolve isso? O erro mais comum é a inversão: pular direto para o sistema sem entender o problema e as pessoas.

## Como aplicar
1. **Repita o problema em voz alta antes de qualquer solução**: "então, o que você está realmente tentando resolver?" Muitas vezes o pedido de ferramenta mascara um problema de processo.
2. **Mapeie todos os stakeholders humanos**: quem precisa aprovar? Quem usa hoje? Quem vai resistir? Quem tem restrições (legal, segurança, contratos)? Salte esse passo e seu sistema vai falhar na implementação.
3. **Só depois do PPS, escolha o sistema**: com problema e pessoas claros, a escolha de sistema se torna óbvia (ou pelo menos muito mais fácil).
4. **Use como filtro de demandas inbound**: quando alguém chegar com "preciso de X ferramenta", redirecione com "me conta qual é o problema que você está tentando resolver".
5. **Aplique em decisões de contratação também**: "o que o candidato está tentando resolver [problema]? Quem está envolvido [pessoas]? Que sistema ou processo precisaria mudar? [sistema]"

## Insights por autor
### [[Austin Hay]]
- "As pessoas geralmente pulam direto para o sistema. Elas dizem: tem esse problema, só precisam resolver com a ferramenta. Mas se você recuar, primeiro entende o problema — o que essa pessoa está tentando resolver? — depois as pessoas envolvidas, depois o sistema."
- Exemplo: gerente de vendas quer automatizar onboarding de novos vendedores. PPS: problema = processo de onboarding manual demorado; pessoas = CRO precisa aprovar, vendedores precisam ser treinados; sistema = automação no CRM. "Quando você entende o problema e as pessoas, fica muito fácil desenhar o sistema para resolver."
- Contexto: criado para technologists de marketing/sistemas, mas aplicável a qualquer decisão de produto ou operações.
