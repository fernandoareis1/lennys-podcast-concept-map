# Framing — Estreitar o Problema antes de Shapear

**Fluxo:** [[01 - Discovery & Pesquisa]] · **Tema:** [[Discovery — Descoberta contínua & validação]] · **Camada:** L3
**Tipo:** Prática · **Fontes:** [[Ryan Singer]]

## Ideia central
Antes de shapear — de criar a solução — é necessário **estreitar o problema**. Ryan Singer chama essa etapa de "framing": o trabalho de negociar e afinar o problema até que ele seja específico o suficiente para ser shapeável.

**O problema do blob expansivo:** quando o problema é "calendar", a sessão de shaping não tem fim. Calendar é um universo. Mas quando o problema é "usuários não conseguem ver os espaços vazios na sua agenda para saber quando estão livres", isso é um problema concreto com limites — e agora é possível pensar em soluções que caibam em 6 semanas.

**Framing não é descoberta de produto completa.** É uma pré-condição específica do processo de shaping: você precisa saber *o que está tentando resolver* de forma específica antes de começar a pensar em como. Se o input do shaping for vago ("build a dashboard"), o shaping vai produzir mais perguntas do que respostas.

**De onde vem o problema:** pode vir de research qualitativo, de conversas com clientes, de pedidos repetidos de vendas, de CEO com ideia no chuveiro. O trabalho de framing não é gerar o problema do zero — é *negociar* esse input até ser específico o suficiente. Qual é o core do pedido? Onde está o valor? O que é o mínimo que precisamos resolver para que a feature valha o tempo?

**A pergunta de framing:** qual é o struggle real que o usuário está tendo? Ryan Singer recomenda o trabalho de JTBD (Bob Moesta) como ferramenta complementar ao framing — identificar o "struggling moment" antes de entrar no shaping.

## Como aplicar
1. **Nunca entre em uma sessão de shaping com "build X".** Primeiro pergunte: qual é o problema específico que X resolve? Para qual usuário? Em qual contexto?
2. **Negocie o problema de volta até ter limites.** "Calendar" precisa virar "usuários perdem reuniões porque não conseguem ver slots livres rapidamente." Isso define: quem (usuário com reuniões), o quê (ver slots), quando (rapidamente).
3. **Pergunte: se só resolvermos esse aspecto, entrega valor?** O framing perfeito encontra o slice de problema que, por si só, justifica o investimento de tempo.
4. **Use o JTBD como ferramenta de framing.** Qual é o job que o usuário está tentando completar? Qual é o struggling moment específico?
5. **O framing pode revelar que não é hora de shapear.** Às vezes, após o framing, fica claro que o problema não é urgente o suficiente ou que você não tem informação suficiente. Isso é um win — melhor agora do que na semana 4 do build.

## Insights por autor
### [[Ryan Singer]]
- "Se o problema for 'calendar', o shaping vai ser uma coisa que se expande para sempre e não vamos conseguir chegar a nada."
- "Conseguimos estreitar até: entendemos que para nossos clientes específicos que pedem isso de novo e de novo, é mais sobre 'eu preciso ver espaços vazios'. Na view de agenda existente, só posso ver coisas já agendadas."
- "Antes da definição do problema, há essa questão de: qual é a demanda? Onde as pessoas estão lutando? É aí que eu alcanço o trabalho do Bob Moesta com JTBD."
- "O shaping é o supply side. O JTBD é o demand side. Framing é onde esses dois lados se encontram."
- "Às vezes, o problema não está no build ou no processo. O input desde o início é muito vago. Não sabemos o que é importante para os clientes ou onde está o valor."
