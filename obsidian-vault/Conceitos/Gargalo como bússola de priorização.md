# Gargalo como bússola de priorização
**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Anton Osika]]

## Ideia central
Priorização reduzida a um algoritmo simples: **identifique o maior gargalo → resolva-o completamente → identifique o próximo**. Não existe roadmap complexo, não existe matriz de pontuação: existe uma pergunta única que se repete. O poder desse método é que força o time a convergir em *um* problema de cada vez, evita a dispersão de esforço em muitas iniciativas paralelas, e é suficientemente simples para ser executado sem um PM formal. Em startups engineering-led, o engenheiro que entende o gargalo técnico muitas vezes também entende qual é a iniciativa de maior impacto — e pode agir sem handoffs.

## Como aplicar
1. **Toda segunda-feira: "qual é nosso maior problema agora?"** — não "o que vamos fazer?", mas "o que está mais nos impedindo de crescer/servir usuários?"
2. **Evite listas de prioridade longas**: se você tem cinco problemas "prioridade 1", você não tem priorização — você tem uma lista de desejos. Force escolha de um.
3. **Dê nome ao gargalo atual para o time inteiro**: na Lovable, a iniciativa de tornar o sistema mais agêntico está explicitamente comunicada como "o maior problema agora." Todo o time sabe.
4. **Revise semanalmente, não mensalmente**: o maior gargalo muda quando você resolve o anterior — semanas são o ciclo certo, não quarters.
5. **Conecte à capacidade técnica**: em produtos onde a solução do gargalo envolve arquitetura técnica, engenheiros devem ser os que identificam — não produto ou design isolados.

## Insights por autor
### [[Anton Osika]]
- "Top line? Identificar qual é o maior gargalo, qual é o maior problema, e iterar rápido: ok, este é o maior problema, vamos realmente resolvê-lo. Depois pegar o próximo. Não pensar demais, não sonhar com o roadmap longo."
- A Lovable opera com planejamento semanal e FigJam board com todos os principais problemas ranqueados por importância e quando focar em cada um.
- Não tem PM formal com 18 pessoas — é engineering-led porque "a solução certa para o problema pode estar entrelaçada em detalhes técnicos."
- Rodadas de "polish week" quando o maior problema é qualidade, não features — gargalo de polish é tão válido quanto gargalo de funcionalidade.
