# Hipótese Fundadora — Tornar o Implícito Explícito
**Fluxo:** [[02 - Estratégia & Visão]] · **Tema:** [[Estratégia — Estratégia de produto]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Jake Knapp & John Zeratsky]]

## Ideia central
Todo produto tem uma hipótese central — a crença de que resolver este problema para este cliente com esta abordagem vai gerar este resultado. O problema: essa hipótese quase nunca é explicitada. Quando fica implícita, é impossível interrogar, testar ou falsificar. Tornar a hipótese explícita transforma opiniões em variáveis testáveis.

"Every new product has at its core a hypothesis. It's just usually not explicit... Because it's not explicit, it's very hard to interrogate it and test it and find out if the different variables in it are the right variables."

## A estrutura Mad Libs da Hipótese Fundadora
"Se resolvermos [problema] para [cliente] com [abordagem], acreditamos que eles vão nos escolher sobre [concorrentes] porque de [diferenciador 1] e [diferenciador 2]."

Cada elemento é uma variável testável:
- **Problema**: o cliente realmente tem esse problema? (confirmar em conversas)
- **Cliente**: esse é o segmento certo? (confirmar quem está "muito decepcionado" se o produto sumisse)
- **Abordagem**: essa é a melhor forma de resolver? (testar com protótipos)
- **Concorrentes**: quem o cliente considera como alternativa? (frequentemente diferente do que o time assume)
- **Diferenciadores**: esses differentiators realmente importam para o cliente? (a mais esquecida)

## O Scorecard de Validação
Com a hipótese explícita, Jake e John criam um scorecard que avalia cada variável em conversas com usuários:
- Vermelho: a variável está errada — o cliente não reconhece o problema / não está nesse segmento / não valoriza o differentiator
- Amarelo: há sinal mas não está claro ainda
- Verde: confirmado com evidência

Sprint a sprint, o objetivo é mover variáveis de vermelho para amarelo para verde. Se uma variável está persistentemente vermelha, a hipótese precisa ser revisada — não o protótipo.

## Por que o implícito é perigoso
Quando a hipótese fica implícita:
1. Times distintos têm hipóteses diferentes em suas cabeças e ficam se contradizendo sem saber
2. Aprendizados de usuários são interpretados de formas diferentes ("esse feedback confirma ou refuta nossa hipótese?")
3. É fácil se enganar — "os usuários gostaram do demo" não diz qual variável da hipótese foi confirmada
