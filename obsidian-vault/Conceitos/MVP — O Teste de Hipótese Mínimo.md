# MVP — O Teste de Hipótese Mínimo

**Fluxo:** [[05 - Experimentação & Dados]] · **Tema:** [[Experimentação — Aprender com pouco e qualitativo]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Eric Ries]]

## Ideia central
MVP (Minimum Viable Product) **não é o produto mais básico possível** — é a forma mais eficiente de obter validação sobre uma hipótese específica. Para qualquer hipótese que você quer testar, qual é o jeito mais rápido e barato de saber se ela é verdadeira? A resposta a essa pergunta é o seu MVP.

A lógica: toda construção passa por um momento em que está "metade pronta". Esse momento sempre vai existir. A questão é: você vai ter clientes usando e te dando feedback nesse ponto, ou vai esperar o "produto completo"?

## Como aplicar
1. Defina a hipótese que você quer testar — não "lançar algo", mas validar uma crença específica.
2. Pergunte: qual é o menor experimento que me diz se essa hipótese é verdadeira?
3. Encontre 10 clientes (não 10 milhões) e deixe-os usar. Ouça o que é horrível.
4. Uma série de experimentos, nunca um só: um único MVP com 0% de conversão não significa nada.
5. Em mercados de alta exigência (jet engines, data centers), o MVP ainda faz sentido — conter o risco de erro ainda compensa.

## Insights por autor
### [[Eric Ries]]
- "MVP não é sobre um produto tosco. É sobre: para a hipótese que estou testando, qual é a forma mais eficiente de obter validação?"
- Maior misconception: "Lean startup means cheap." Não significa — LTSE levou 10 anos para construir o MVP.
- Na IMVU: avatars que não andavam (sem inverse kinematics) eram "feios demais para mostrar". Um hack simples — teleporte sem animação — foi o MVP que testou se os usuários se importavam com movimento vs. colaboração.
- "Startups fazem produtos tomarem meses ou anos. Durante esse tempo, você já tem half-built products. Não usar esse estágio para aprender é um desperdício."
