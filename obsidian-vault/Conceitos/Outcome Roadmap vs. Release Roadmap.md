# Outcome Roadmap vs. Release Roadmap
**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Estratégia vs execução]] · **Camada:** L3
**Tipo:** Insight · **Fontes:** [[Itamar Gilad]]

## Ideia central
O tipo de roadmap que você usa determina se sua equipe vai aprender ou apenas executar. **Release roadmaps** matam o aprendizado porque criam pressão para lançar uma feature específica numa data específica. **Outcome roadmaps** direcionam para o resultado, liberando o time para encontrar o melhor caminho.

## A distinção

### Release Roadmap (problemático)
- "Q3: lançar feature X"
- "Outubro: lançar Y"
- Uma vez que a equipe sabe que o objetivo é lançar X até outubro → aprendizado para
- Pressão de datas força compromisso com a solução antes de entender se ela funciona
- Elimina a capacidade de pivotar mesmo quando evidências mostram que a feature não vai resolver o problema

"Se você faz isso e as pessoas sabem que o objetivo é lançar aquela coisa até outubro — esqueça aprendizado, esqueça evidence-guided."

### Outcome Roadmap (recomendado)
- "Outubro: alcançar este outcome" — ex: mover retenção de D30 em X pontos
- "Q4: lançar em 3 novos países" — outcome geográfico, não feature específica
- "Aumentar uso na Índia em Y%" — resultado mensurável, não lista de entregáveis
- A equipe decide como alcançar o outcome → aprendizado e adaptação possíveis

## Quando é OK ter datas fixas de feature
Apenas quando a ideia já passou por validação suficiente e está na zona vermelha do [[ICE Score e o Medidor de Confiança]]. Se você tem alta confiança (experimentos, betas, dados sólidos), comprometer-se a uma data de lançamento é razoável. O problema é quando há comprometimento de data com baixa confiança.

## Ligação com o framework GIST
No [[GIST — Camadas do Desenvolvimento de Produto]], os **Steps** (passos) são marcos de aprendizado, não marcos de feature. Cada passo tem como output aprendizado + evidência, não um lançamento. O roadmap que suporta GIST é necessariamente um roadmap de outcomes — porque o que você vai construir em cada passo depende do que aprendeu no anterior.

## Por que funciona no GIST Board
"Cada vez que você roda um passo — se foi bem-sucedido — você tem evidência e pode voltar aos managers e contar... a discussão se torna muito mais matizada e objetiva."
