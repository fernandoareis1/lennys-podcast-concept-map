# Priorizar o IC sobre o Gestor — Anti-Bloat

**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** mindset · **Fontes:** [[Nan Yu]]

## Ideia central
A principal causa de bloat em software B2B é features pedidas por gestores médios para facilitar reporting — ao custo de tornar o workflow dos ICs (Individual Contributors) pior. A Linear tem uma política explícita: features que satisfazem gestores ao custo dos ICs são "no por padrão" — sem debate. Isso parece arriscado do ponto de vista comercial (perder deals), mas na prática é o oposto: quando os ICs não adotam o produto, os dados ficam errados (campos marcados aleatoriamente), e o gestor nunca consegue o reporting que queria. Priorizar o IC não é altruísmo — é a única forma de o produto funcionar de verdade.

## Como aplicar
- Identifique as categorias de pedido que "sempre" resultam em pior experiência para o usuário final — coloque-as em uma lista de "não fazemos" sem necessidade de debate caso a caso.
- Para pedidos de gestores, pergunte: "Como esse feature vai afetar quem realmente usa o produto diariamente?"
- Tente convencer compradores que as features que eles querem são a causa do problema que eles têm (dados ruins), não a solução.
- Quando um pedido parece ser "campo customizado para X", sempre pergunte: qual é o problema real subjacente? Frequentemente há uma solução melhor que não sacrifica a experiência do IC.
- Resista à pressão de vendas para adicionar features de gestão que ICs vão ignorar.
