# Qualidade-Features-Deadline — Escolha Dois

**Fluxo:** [[03 - Priorização & Roadmap]] · **Tema:** [[Priorização — Priorização & foco]] · **Camada:** L3
**Tipo:** Framework · **Fontes:** [[Dylan Field]]

## Ideia central
Para qualquer lançamento novo: **Qualidade · Features · Deadline — escolha dois**. É impossível ter os três ao mesmo tempo sem comprometer algum deles. Esta é uma ferramenta de decisão explícita, não uma desculpa para não ter qualidade.

**Os três cenários:**
- **Qualidade + Features**: acontece quando o prazo é flexível (ideal para lançamentos fundacionais)
- **Qualidade + Deadline**: acontece quando o escopo de features é reduzido (entregas menores, mais polidas)
- **Features + Deadline**: acontece quando você aceita iterar na qualidade depois (só válido se há loop de iteração garantido)

**Nuances importantes:**
- Software pode melhorar iterativamente — diferente de produto físico, não precisa sair perfeito
- Mas você não pode sempre adiar qualidade para depois — hábitos e reputação se formam nas primeiras impressões
- Existe um "mínimo de qualidade" para cada lançamento — abaixo dele não vale lançar mesmo com features corretas

**Aplicação:**
Definir explicitamente, antes de começar, qual dos três vai ser sacrificado neste lançamento específico. Torna o trade-off consciente e evita que a equipe otimize pela métrica errada.

## Como aplicar
1. No início de qualquer projeto, declare explicitamente: "Nesse lançamento, vamos priorizar _____ e _____, e aceitar sacrificar _____"
2. Se a resposta for "Features + Deadline", garanta: existe um plano de iteração real para qualidade? Ou é desculpa?
3. Defina o "mínimo de qualidade" para cada entrega — seja específico: o que exatamente não pode estar abaixo da barra?
4. Revise essa escolha a cada milestone — mudanças de contexto podem mudar a prioridade entre os três
