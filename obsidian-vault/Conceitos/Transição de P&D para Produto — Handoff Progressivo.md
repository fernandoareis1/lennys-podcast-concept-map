# Transição de P&D para Produto — Handoff Progressivo

**Tema:** [[Liderança — Liderança & founder]] · **Fluxo:** [[09 - Liderança & Times]]

O maior risco de incubar um produto em uma equipe de P&D e depois "passá-lo" para um time de produto é exatamente o handoff: quem sabe fazer o produto vai embora antes de quem vai mantê-lo ter aprendido o suficiente. Ryan J. Salva coordenou a transição do GitHub Next (P&D) para o time de EPD (Engineering, Product, Design) no GitHub Copilot e formalizou os princípios que tornaram esse handoff progressivo.

## O processo

A transição do Copilot não foi uma entrega — foi uma transferência gradual:
1. **Identificação do sinal real**: GitHub Next detectou que desenvolvedores estavam tendo reações "mind-blown". Esse foi o gatilho para envolver mais o time EPD.
2. **Pesquisadores como semente do novo time**: Alguns pesquisadores do GitHub Next foram temporariamente movidos para criar um "EPD squad". Eles não saíram do projeto — eles tornaram o projeto possível para o novo time.
3. **Substituição em seat**: O critério para o pesquisador retornar ao P&D não é um calendário — é **encontrar um substituto que tenha absorvido todo o conhecimento necessário**.
4. **Processo de volta gradual**: Os pesquisadores começaram a voltar para o GitHub Next apenas quando múltiplos EPD squads já haviam assumido o produto e o roadmap.

## Os 3 princípios

> *"The criteria for moving researchers back into their R&D team can't be based on a calendar. It needs to be based on a replacement in seat."*

1. **O time de produto deve sentir que controla o roadmap** — se o roadmap continua sendo delegado ao time de P&D, o time de produto nunca terá ownership real.
2. **Engineering fundamentals são os contratos** — o que diferencia P&D de produto operacional é o processo de qualidade (segurança, privacidade, uptime, acessibilidade). Isso parece "não natural" para pesquisadores no início.
3. **Mix deliberado de perfis** — o time que transiciona precisa de pesquisadores (que conhecem a visão) E engenheiros que são confortáveis mantendo um serviço em produção.

## Aplicação para PMs e líderes

- Antes de "passarem" um projeto de R&D, perguntem: quem no time receptor já absorveu o domínio?
- Não façam o handoff por data — façam por competência demonstrada
- O time de P&D deve ficar disponível por um tempo sobrepassado (overlap), não desaparecer no dia da "entrega"
- A inovação não deve se concentrar só no R&D — o time de produto precisa internalizar que o roadmap é deles
